關於我

企業管理學系背景，現為 AI 解決方案方向的工程實習生。

我的位置不在「最強的工程師」，也不在「只談策略的人」，而是兩者之間那座橋：能把一個模糊的商業問題，端到端做成可落地的 AI 方案，並且習慣用成本與工時去衡量它值不值得。

從資料清洗、模型導入到後端 API 串接，能獨立跑完一輪；而企管的訓練，讓動手之前會先問「這個問題真正的瓶頸在哪、AI 是不是對的解」。

正在尋找能把這套跨域能力用上的位置——在推動 AI 轉型、為各個職能導入 AI 以提升效率的團隊裡，當那個能在商業與技術之間翻譯、並讓專案真正落地的人。

1

智慧驛站包裹出庫稽核系統

以視覺語言模型取代人工客服的取貨稽核判斷，並在客戶反映成本後，透過模型替換與提示工程把推論成本降約七成。
▾
痛點

無人取貨站要確認「取貨者有沒有真的到掃描儀掃描才離開」，原本仰賴人工客服盯著畫面判斷，難以規模化；而真實人流的動線與遮擋，也讓純規則難以涵蓋。

方案

先以 Gemini 視覺語言模型搭配電子圍籬兩階段狀態機，自動完成判斷、取代人工客服——Stage 1 在貨架圍籬偵測取貨、Stage 2 在掃描儀圍籬偵測掃描，物件偵測抓出人的座標後自寫座標碰撞判定，再交由 VLM 做語意確認。客戶反映 API 成本偏高後，改用 Qwen-VL，重新進行提示工程並以 45 個場景重測，綜合成本與成效選出最適模型。

關鍵技術決策
  • 成本優化：把 VLM 換成輕量 8B 模型，並在 Stage 2 先裁切出掃描儀區域，讓小模型聚焦小物件、解決解析度不足——更準且更省。
  • 穩定度：監視器視角下人物偵測會斷續，加入「允許漏抓不中斷計時」的容錯與 LLM 冷卻機制，避免重複觸發。
  • 備援設計：另實作語音聲紋比對，定位為 VLM 故障或網路異常時，輔助現場客服判斷的備援路徑。
商業成果

單筆交易推論成本約 TWD 0.0035；換用輕量模型後較原 Gemini 方案約降 73%。在 45 個有效測試場景上完成驗證並產出測試報告。

2

合約風險分析工具

先做模型選型方法論，再用「快模型全文 + 本機條號回填」把大型合約分析從分鐘級且不穩，壓到 20–60 秒可用。
▾
痛點

法務人員判定合約風險時，得逐條讀完動輒五萬字的合約，耗時又容易遺漏；而把整份合約一次送進分析服務，又會撞到約 100 秒的閘道逾時（504），模型也常把風險條款的條號標錯位置、影響可信度。

方案

系統性評測 10 個大型語言模型在「全文單次」與「分段彙整」兩種策略下的表現，整理時間、條號命中與風險等級分佈，據此選出最佳解：快模型全文單次 + 逾時自動重試，搭配純本機的「條號回填校驗」。前端是一頁風險儀表板。

關鍵技術決策
  • 用數據說話：證明快模型全文（20–60 秒）優於慢模型分段（3.5–23 分鐘，且彙整步驟必逾時導致風險等級偏高）。
  • 條號回填：用加權字元 n-gram 把模型自報的條號回原文比對，只有在別條證據明顯較強時才覆寫，零 API 成本、不破壞正確項，實測 10/10 修正。
  • 自建本機 CORS 代理，解決瀏覽器無法直連後端的問題。
商業成果

把整份動輒五萬字的合約，壓縮成標明風險條文位置的表格摘要，讓法務能直接跳到風險條文查看，大幅降低逐條閱讀的時間；條號校驗純本機、不增加任何 API 費用。

3

Telegram 產品 Bug 回報機器人

把分散的 Bug 回報收斂成一條閉環——引導式填單、自動歸檔、分流通知、結案回拋，並附完整交接文件。
▾
痛點

測試人員回報 Bug 散落在訊息、信件與表格之間；工程端追蹤狀態、回拋通知給原回報者的環節經常斷裂。

方案

用有限狀態機（FSM）做引導式填單，逐步收集回報；自動依產品分流寫入 Google Sheet、圖片上傳圖床、推播工程群組並附上該產品負責人；工程端更新狀態後，由 Apps Script 監聽變更、自動回拋通知原回報者，形成完整閉環。

關鍵技術決策
  • config-driven 設計：產品、模塊、負責人、白名單都放在設定分頁，新增產品免改程式碼，並支援 /reload 不重啟熱重載。
  • 權限取捨：圖片因雲端硬碟權限政策卡關改用圖床；通知腳本因簡單觸發器無權呼叫外部請求，改用安裝式觸發器。
  • 關注點分離（SoC）：程式依設定、資料模型、服務與對話流程分為四層，維護時定位明確。
商業成果

一個完整可交接的產品：附系統架構圖、維護手冊、從零建置教學、v1→v2 演進紀錄與已知限制清單，接手者能直接上手。

4

產線螺絲組裝防錯系統

8 個工作天全流程獨立交付的產線防錯 PoC，並沉澱出一套可複用於工業偵錯的 FSM 範例與 SOP。
▾
痛點

螺絲組裝目前沒有自動檢查機制，一旦漏放或錯放，整個料盤就報廢，是良率與成本的痛點；而實體廠房相機會物理位移、手部會遮擋料盤，固定座標的判定方式很快就會失準。導入 AI 與自動化偵錯，可即時攔截錯誤、提升良率並降低報廢花費。

方案

自訓 YOLO 偵測料盤上的螺絲、另訓手部物件偵測模型，搭配三階段狀態機（等待取料 → 進入料盤 → 離開結算），比對「取了哪個料盒」與「料盤實際多了哪個孔位的螺絲」是否一致，不一致即報警。

關鍵技術決策
  • 用相位相關性（phase correlation）計算料盤的物理位移，把當前偵測座標還原回基準座標系再比對孔位，解決相機位移問題。
  • 手部模型初版只標了料盤上的手，導致拿料盒時偵測不到；重新補標「無手／拿料」情境後重訓修正。
  • 同一套狀態機方法論後續複用到另一條 PCB 板產線（雙電子圍籬），驗證了可移植性。
商業成果

全流程獨立交付：從客戶提供的實際組裝影片出發，轉影格、裁切料盤，標記並訓練判斷孔位是否有螺絲的 YOLO，再標記並訓練偵測手部的 YOLO，最後串接狀態機完成防錯系統；約 650 筆資料訓練、模型表現良好。

5

CAD 工業圖檔版本差異比對

把百萬級工業圖檔轉成可比對的結構化資料，用幾何指紋 + 集合運算 4 分鐘找出改版差異。
▾
痛點

工程圖改版後需要逐版檢核差異。AutoCAD 雖內建版本比對功能，但面對肥大的工程圖檔速度極慢；檔案又大到無法直接餵給 LLM 或即時渲染比對，改版檢核因此成為效率瓶頸。

方案

自寫 dwg→dxf→csv 轉檔管線，用 ezdxf 把巢狀圖塊「炸開」、曲線離散化、多段線拆解，壓平成關聯式表格；再為每個物件建立幾何指紋（Hash），用集合運算算出新增、刪除、不變，最後以紅綠對比圖視覺化。

關鍵技術決策
  • 放棄「文字摘要餵 LLM」與「渲染成圖比對」兩條路（前者失真、後者百萬物件下太慢），改走指紋比對——這是規模可行的唯一解。
  • 指紋座標取小數點後四位，消除浮點誤差造成的假差異。
  • 釐清解析套件對不同物件型別的迭代邏輯不同（曾導致 114 萬物件只抓到 6 個的坑）。
商業成果

4 分鐘內解析 1,538 萬個幾何物件、捕獲率 99.7%、全程無記憶體溢出；大幅縮減人工比對時間（估計值，基於人工比對流程分析）。

6

小模型 OCR 提示詞工程

摸清小型 OCR 模型的能力邊界，為四類文件各設計穩定、可被後端直接解析的輸出格式。
▾
痛點

企業把紙本文件電子化時，發票、名片、表格這類文件版式雜亂、難以自動解析，往往得人工逐筆 key in，耗時又容易出錯。

方案

針對文檔、名片、表格、發票四類，順應模型天性只做「順勢轉錄」，各自設計極簡提示詞與最適輸出格式（文檔用 Markdown+HTML 表格+LaTeX、名片用扁平 JSON、表格用 HTML、發票用 JSON 或純文字雙方案），並撰寫後端交接規格。

關鍵技術決策
  • 用正面規則與輸出 Schema 取代負面禁令，並要求結構一致性，讓後端能直接以程式解析。
  • 複雜財報表格會讓小模型崩潰，提出由後端先用 OpenCV 智慧裁切（找行距）再餵模型的策略。
  • 加上防呆字串攔截，避免模型回傳的固定錯誤訊息讓後端崩潰。
商業成果

交付一套後端可直接載入解析的穩定格式與規格，讓資料不需人工干預就能進資料庫。以發票報帳為例，過去要在財務系統人工填單，現在上傳發票即自動提取關鍵欄位；名片則能幫業務快速建立聯絡簿，不怕名片遺失。歷經五十多版提示詞迭代收斂。
